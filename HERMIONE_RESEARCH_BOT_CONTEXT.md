# Контекст переезда Hermione Research Bot

Этот документ нужен как переносимый контекст для создания нового GPT/Codex-проекта
под Гермиону: локального Telegram-сервиса для глубоких веб-исследований на базе
идей среды выполнения OpenClaw.

## Пользователь и правила коммуникации

- Обращайся к пользователю: Илья.
- Пользовательский язык: русский.
- Тон: прямой, практичный, без лишнего шума.
- Общая задача: делать правильные, удобные, сбалансированные и платформенные
  инструменты, которые можно постепенно развивать в среде с legacy- и
  modern-проектами.
- Следуй DevEx-принципам из OpenClaw/AGENTS-контекста:
  постепенное внедрение, учет legacy, разноуровневая человеко- и
  машино-читаемая документация, расширяемость и контролируемая кастомизация,
  crash/bug/fail-репорты внутренних ошибок, внимание к производительности и
  баланс расхода ресурсов.

## Замысел продукта

Гермиона — не универсальный чат-бот.

Гермиона — ассистент только для глубоких профессиональных веб-исследований:

- формулировать план исследования;
- раскладывать вопрос на подвопросы;
- намеренно искать доказательства и контрдоказательства;
- читать реальные источники;
- фактчекать числа, даты, имена и причинно-следственные утверждения;
- отслеживать противоречия и неопределенность;
- самостоятельно решать, куда продолжать исследование;
- останавливаться, когда дополнительный поиск уже не меняет решение существенно;
- выдавать полезный для принятия решения отчет с источниками, уверенностью,
  оговорками и явными ограничениями.

Бот нужен для долгих, объективных, пригодных для решений исследований, а не для
быстрых Q&A-ответов.

## Принятые решения

### Область применения

- Начальная область: только веб-исследования.
- Не входит в начальную область:
  - исследование локальных репозиториев;
  - исследование внутренних документов;
  - PAF/CPO-методологический аудит;
  - поведение универсального ассистента;
  - произвольные coding-задачи;
  - автономные внешние write-действия.

### Пользовательский сценарий

- Интерфейс: Telegram-чат.
- Стиль среды выполнения: асинхронные исследовательские задачи.
- Пользователь отправляет исследовательский вопрос.
- Гермиона подтверждает задачу, запускает job, отправляет обновления прогресса
  и позже присылает финальный отчет.
- Долгие задачи не должны блокировать процесс Telegram polling.

### Глубина исследования

Режим по умолчанию: адаптивное глубокое исследование.

Смысл:

- Гермиона по умолчанию исследует глубоко.
- Она продолжает, пока новые источники меняют вывод, выявляют значимые
  противоречия, закрывают важные пробелы или повышают уверенность в решении.
- Жесткий лимит защищает от бесконтрольного расхода времени и денег, например
  60 минут на один цикл.
- Если важные пробелы остаются, Гермиона останавливается, объясняет, что уже
  известно, что остается неопределенным, и предлагает сфокусированный план
  продолжения.
- `/continue` запускает следующий исследовательский цикл с учетом предыдущего
  отчета, незакрытых пробелов и списка уже использованных источников.

Этот режим предпочтительнее безлимитного исчерпывающего поиска, потому что
веб-исследование иначе может деградировать в повторяющиеся источники, лимиты
поисковиков, боковые ветки с низкой ценностью для решения и неконтролируемый
расход ресурсов.

## Рекомендуемая архитектура

Рекомендуемый вариант: Telegram-среда выполнения OpenClaw + Searcharvester как
соседний локальный сервис.

### Почему этот вариант

- Самый быстрый путь к рабочему локальному research-боту.
- Переиспользует Searcharvester для поиска, извлечения страниц и глубокого
  исследовательского цикла.
- Оставляет Telegram UX, персону, правила поведения, прогресс и операционные правила
  Гермионы в ее собственном репозитории.
- Позволяет постепенно заменить или усилить внутренности Searcharvester позже.
- Не требует на старте vendor-копии стороннего AGPL-проекта в кодовую базу
  Гермионы.

### Общая форма

Репозиторий Гермионы:

- Telegram-среда выполнения;
- `SOUL.md` Гермионы / системный промпт только для исследований;
- Telegram-команды и UX для исследовательских задач;
- код адаптера, который вызывает Searcharvester HTTP API;
- примеры конфигов без секретов;
- runbook;
- тесты;
- документация.

Searcharvester:

- запускается локально как соседний сервис;
- предоставляет `/search`, `/extract`, `/research`;
- закрепляется через конкретный Docker image или git commit;
- на старте не копируется внутрь репозитория Гермионы.

Рекомендуемая структура папок:

```text
Projects/Work/
  Salamander/
  HermioneResearchBot/
```

У каждого бота должен быть свой `.git`-репозиторий. Не нужно класть две
несвязанные Git-истории в одну рабочую папку. Не стоит использовать разные
ветки одного репозитория как разные продукты.

## Что известно про Searcharvester

Изученный репозиторий:

- `https://github.com/vakovalskii/searcharvester`
- изученный commit: `376c11f72f0c55f215cafd4491cd4690b16e942b`
- дата commit, увиденная локально: `2026-04-24 14:07:03 +0300`
- лицензия, увиденная через GitHub API: `AGPL-3.0`

Полезные части:

- `POST /search`: Tavily-совместимый поиск через SearXNG.
- `POST /extract`: URL в чистый markdown через `trafilatura`, с пагинацией.
- `POST /research`: асинхронная исследовательская задача, которая возвращает markdown-отчет
  с цитатами.
- `GET /research/{job_id}`: статус job и финальный отчет.
- `GET /research/{job_id}/events`: SSE-поток событий.
- `GET /research/{job_id}/snapshot`: полный снимок событий.
- Есть React UI для прогресса и отладки исследовательских задач.

Важная поправка к публичному описанию:

- В публичном посте описаны эфемерные Hermes-контейнеры на задачу и
  `docker-socket-proxy`.
- В текущем репозитории архитектура уже изменилась.
- Текущий `docker-compose.yaml` говорит, что adapter запускает `hermes acp` как
  subprocess внутри контейнера `tavily-adapter`.
- Это проще запускать, но слабее как модель изоляции.

Методика исследования Searcharvester:

- Раунд 1: 2-3 исследовательских субагента параллельно.
- Между раундами: ведущий агент готовит summary исследователей и список фактов для
  проверки.
- Раунд 2: критик + фактчекер, уже с контекстом Раунда 1.
- Финальный синтез: markdown-отчет с источниками, оговорками и разбором
  разногласий, если они есть.

Риски и шероховатости:

- Лицензия AGPL требует аккуратности, если сервис когда-нибудь станет
  распространяемым или корпоративным.
- Документация частично отстает от текущей архитектуры.
- Текущая модель изоляции слабее старой модели с отдельным контейнером на задачу.
- SearXNG может упираться в блокировки поисковиков и капчи.
- `/extract` cache живет в памяти и имеет TTL.
- Состояние задач в основном in-memory плюс файлы; устойчивую очередь и resume нужно
  отдельно проверить и усилить.
- В изученном репозитории `hermes-data/config.yaml` содержал custom `base_url`;
  в Гермионе это должно быть явно настроено и не должно молча наследоваться.
- Для production-доступа нужны auth, TLS, лимиты запросов, лимиты ресурсов и аккуратная
  работа с секретами.

## Поведенческий контракт Гермионы

Гермиона должна по умолчанию работать по принципу «сначала доказательства».

Правила:

- Не отвечать на исследовательские вопросы из памяти, если задача помечена как исследование.
- Всегда разделять факты, утверждения с опорой на источники, интерпретации и
  рекомендации.
- Намеренно искать контрдоказательства.
- По возможности проверять важные факты минимум через два независимых домена.
- Предпочитать первичные источники, официальную документацию, стандарты,
  filings, научные статьи и надежные публикации.
- Явно показывать неопределенность.
- Явно показывать, если источники расходятся.
- Давать ссылки на источники для значимых утверждений.
- Не скрывать неполноту исследования, если остаются важные пробелы.
- Не выполнять внешние write-действия.
- Не собирать приватные или чувствительные персональные данные сверх легитимного публичного
  профессионального контекста.

## Предлагаемые Telegram-команды

Минимальный стартовый набор:

- `/start`: объяснить исследовательское назначение Гермионы и как поставить задачу.
- `/research <question>`: запустить новую адаптивную глубокую исследовательскую задачу.
- `/status`: показать текущую активную задачу и фазу.
- `/cancel`: остановить текущую задачу.
- `/continue`: продолжить от незакрытых пробелов последнего отчета.
- `/settings`: показать текущие ограничения, например max duration,
  max sources, model и search backend.

Опционально позже:

- `/sources`: показать источники, уже использованные в текущей задаче.
- `/brief`: отправить компактное summary текущего состояния до финального отчета.
- `/export`: экспортировать отчет в markdown.
- `/mode`: если позже появятся режимы, но начальный default должен оставаться
  глубоким.

## Формат отчета

Отчет по умолчанию должен быть полезен для принятия решения, а не просто
описательным.

Рекомендуемые разделы:

```text
# Заголовок

## TL;DR
Короткая выжимка, ориентированная на решение.

## Исследовательский вопрос и контекст решения
Что именно исследовалось и какое решение это должно поддержать.

## Метод
Стратегия поиска, типы источников и подход к фактчекингу.

## Находки
Ключевые выводы с опорой на источники и inline-ссылками.

## Контрдоказательства и разногласия
Что может быть неверным, спорным, устаревшим или контекстно-зависимым.

## Таблица фактчекинга
Важные числа, даты, имена, источник A, источник B, вердикт.

## Вывод для принятия решения
Что выглядит достаточно надежным для действия, что остается неопределенным и почему.

## Уверенность
Высокая/средняя/низкая с причинами.

## Оставшиеся пробелы
Что еще требует исследования.

## Рекомендуемый следующий research-цикл
Только если полезно: сфокусированные следующие вопросы для `/continue`.

## References
Нумерованный список URL.
```

## Предлагаемый первый план реализации

Не нужно реализовывать все сразу. Лучше строить слоями.

### Фаза 1 - каркас нового репозитория

- Создать `HermioneResearchBot` как новый Git-репозиторий.
- При необходимости переиспользовать паттерны текущей Telegram-среды выполнения OpenClaw.
- Убрать понятия, специфичные для PAF-аудитора.
- Переименовать предметный язык в Hermione/research.
- Добавить `.env.example`, README, runbook и тесты.

### Фаза 2 - Searcharvester как соседний сервис

- Добавить локальный конфиг для `SEARCHARVESTER_URL`.
- Добавить HTTP client module:
  - создать исследовательскую задачу;
  - опрашивать задачу;
  - отменять задачу;
  - получать отчет;
  - опционально подписываться на события или читать их.
- Не копировать код Searcharvester внутрь репозитория на старте.

### Фаза 3 - Telegram UX для исследовательских задач

- `/research` запускает задачу и сохраняет состояние активной задачи.
- `/status` сообщает текущую фазу.
- `/cancel` отменяет задачу, если backend это поддерживает.
- Финальный отчет отправляется Telegram-friendly chunks или markdown-вложением.
- Метаданные последней задачи сохраняются для `/continue`.

### Фаза 4 - Soul и правила Гермионы

- Написать `SOUL.md` / системный промпт для research-only поведения.
- Добавить правила качества источников.
- Добавить правила фактчекинга и контрдоказательств.
- Добавить явные правила остановки для адаптивного глубокого исследования.
- Добавить шаблон отчета.

### Фаза 5 - качество и эксплуатация

- Тесты на разбор команд, жизненный цикл задач, нарезку сообщений и обработку ошибок.
- Runtime caps:
  - максимальная длительность задачи;
  - максимальное число одновременных задач;
  - максимальный размер фрагмента Telegram-сообщения;
  - интервал опроса;
  - лимиты размера источников и отчета.
- Логирование и crash reporting.
- Runbook для локального запуска, остановки и обновления.

## Важные ограничения реализации

- Не коммитить секреты.
- Не включать `.env` или Telegram token в Git.
- Не выставлять локальный API наружу без auth/TLS/rate limiting.
- Не менять production dependencies без явного review.
- Закрепить Searcharvester через image tag или commit.
- Держать Hermione и Salamander в отдельных репозиториях, если нет явного
  решения о monorepo.
- Не смешивать PAF-auditor поведение с default prompt Гермионы.

## Открытые вопросы для нового проекта

Их можно решить уже после создания нового проекта:

- Точное имя репозитория: `HermioneResearchBot`, `Hermione` или другое.
- Переиспользовать текущий OpenClaw code копированием выбранных файлов или начать
  с чистого каркаса.
- Как закрепить Searcharvester: Docker image tag, commit checkout или fork.
- Жесткий лимит по умолчанию на один исследовательский цикл: вероятно 60 минут.
- Целевой объем источников по умолчанию: вероятно 15-30 полезных источников, но
  adaptive stopping важнее фиксированного числа.
- Делать обновления прогресса частыми или компактными.
- Отправлять финальные отчеты текстовыми chunks, markdown-файлами или обоими
  способами.
- Поддерживать несколько одновременных задач или одну активную задачу на пользователя.

## Первый prompt для нового GPT/Codex-проекта

Используй этот prompt в новом проекте:

```text
Илья хочет создать Hermione Research Bot: локальный Telegram-сервис на базе
идей OpenClaw, но полностью заточенный под глубокие веб-исследования.

Первая область:
- только исследования, не универсальный ассистент;
- только веб-исследования;
- асинхронные задачи в Telegram;
- режим по умолчанию: адаптивное глубокое исследование;
- жесткий лимит на исследовательский цикл, затем `/continue`;
- итог: отчет для принятия решения с источниками, фактчекингом, контраргументами,
  неопределенностью, confidence и рекомендацией для решения.

Рекомендуемая архитектура:
- отдельный Git-репозиторий для Hermione;
- паттерны Telegram-среды выполнения OpenClaw можно переиспользовать;
- Searcharvester подключить как локальный соседний HTTP backend;
- не копировать код Searcharvester внутрь repo на старте;
- закрепить Searcharvester image/commit;
- вынести Hermione `SOUL.md`, правила исследования, Telegram UX, adapter client,
  docs/runbook/tests в собственный repo.

Начни с проектирования структуры нового repo и минимального инкремента:
запустить Telegram runtime, принять `/research`, создать job в Searcharvester,
показать статус и вернуть отчет.
```
